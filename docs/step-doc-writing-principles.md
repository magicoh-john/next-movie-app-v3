# Step 문서 작성 원칙

step-01 작성 과정에서 발견한 교육 문서 작성 원칙.
step-XX 문서를 새로 작성하거나 검토할 때 아래 체크리스트를 순서대로 확인한다.

---

### 1. 핵심 파일을 소개했으면 즉시 코드가 나와야 한다

"이 단계의 핵심은 X입니다"라고 했으면 X의 코드와 설명이 바로 다음에 나와야 한다.
다른 섹션을 먼저 진행하고 뒤에서 X를 다루면 안 된다.

### 2. 파일 생성/수정 여부를 섹션 첫 줄에 명시한다

```
✅ **`src/components/Providers.tsx` 파일을 새로 생성합니다.**
✅ **`src/app/layout.tsx` 파일을 수정합니다. (create-next-app이 자동 생성한 파일)**
```

### 3. 브라우저에서 결과를 볼 수 있는 것을 먼저 만든다

타입 정의·유틸리티 함수처럼 화면에 영향이 없는 파일은 브라우저 확인 이후에 배치한다.
동기부여를 위해 눈에 보이는 결과가 먼저 나와야 한다.

### 4. 테스트는 시각적 변화가 없어도 의미 있게 설계한다

"Header가 보이면 완료"처럼 이전 단계와 차이 없는 확인은 불충분하다.
해당 단계에서 추가된 기능(TMDB 연결, API 함수 등)을 실제로 실행해서 결과를 눈으로 확인하는 테스트를 반드시 포함한다.
예) tmdb.ts를 만들었으면 page.tsx에서 임시로 데이터를 가져와 화면에 출력해본다.

### 5. 당위성(왜)이 코드보다 먼저 나온다

코드를 보여주기 전에 반드시:
1. 왜 이 파일이 필요한가
2. 왜 이 위치에 있는가
3. 왜 이렇게 작성하는가

를 먼저 설명한다. "이렇게 하세요"만 있고 이유가 없으면 안 된다.

### 6. 잘못된 방법 vs 올바른 방법을 나란히 보여준다

개념 설명보다 코드 비교가 더 직관적이다.

```tsx
// ❌ 잘못된 방법 — 이유
// ✅ 올바른 방법 — 이유
```

### 7. 파일 간 연결 관계를 ASCII 트리로 시각화한다

파일이 서로 어떻게 import되고 감싸지는지 구조를 보여준다.

```
layout.tsx
└── <Providers>
    ├── <Header />
    └── {children}
```

### 8. 이전 프로젝트 비교는 하지 않는다

Vite, movie-app 등 이전 프로젝트와의 비교는 혼란을 준다.
Next.js 자체 개념으로만 설명한다.
비교표가 필요하다면 "Next.js 내부" 개념끼리 비교한다.

### 9. Windows 환경 명령어를 검증한다

- 경로 구분자: cmd에서는 `\`, PowerShell/Git Bash에서는 `/`
- `mkdir` 다중 인자: cmd는 공백 구분 + 백슬래시, PowerShell은 콤마 구분
- shadcn 등 라이브러리가 자동 생성하는 폴더를 미리 파악하고 중복 생성을 피한다

### 10. 코드 주석은 WHY 중심으로 단다

WHAT(코드가 무엇을 하는가)은 코드 자체로 읽힌다.
주석은 WHY(왜 이렇게 했는가), 특히 함정·제약·비직관적인 결정에만 단다.
